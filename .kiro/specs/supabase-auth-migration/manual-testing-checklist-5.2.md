# データアクセス動作確認チェックリスト（タスク 5.2）

## 概要

このドキュメントは、Supabase Auth移行後のデータアクセスとRLSポリシーの動作確認を行うための手動テストチェックリストです。

**対応要件**: 5.1, 5.2, 5.3

**テスト目的**:
- 認証済みユーザーがノートを作成・取得できることを確認
- RLSポリシーが正しく機能し、ユーザー固有のデータ分離が保証されることを確認
- 未認証ユーザーのアクセス制御が正しく機能することを確認

---

## 事前準備

### 必要な環境

- [ ] フロントエンドが起動している（`npm run dev`）
- [ ] Supabase Authが正しく設定されている
- [ ] 2つ以上のGoogleアカウントが利用可能（マルチユーザーテスト用）
- [ ] ブラウザの開発者ツールが開いている（コンソールログ確認用）

### テストデータの準備

- [ ] テスト用のチャンピオンデータが利用可能
- [ ] 既存のノートデータをクリアまたは把握している

## テストケース1: 認証済みユーザーのノート作成

**目的**: 認証済みユーザーがノートを作成できることを確認（要件 5.1, 5.2）

### 手順

1. **ログイン**
   - [ ] ブラウザで`http://localhost:3000`にアクセス
   - [ ] Navbarの「ログイン」ボタンをクリック
   - [ ] Google OAuth認証画面が表示される
   - [ ] テストユーザーA（例: user-a@gmail.com）でログイン
   - [ ] ログイン成功後、Navbarにユーザーアバター画像が表示される

2. **ノートページへ移動**
   - [ ] `/notes`ページに移動
   - [ ] ページが正常に表示される（リダイレクトされない）
   - [ ] ノート作成フォームが表示される

3. **ノート作成**
   - [ ] 「自分のチャンピオン」を選択（例: Ahri）
   - [ ] 「相手のチャンピオン」を選択（例: Zed）
   - [ ] ルーン、アイテム、スペルを設定
   - [ ] 戦略メモを入力（例: "レベル6以降は距離を保つ"）
   - [ ] 「保存」ボタンをクリック

4. **作成確認**
   - [ ] 成功メッセージが表示される
   - [ ] ページがリロードされる、または一覧が更新される
   - [ ] 作成したノートが一覧に表示される
   - [ ] ノートの内容が正しく表示される（チャンピオン画像、メモ等）

5. **データベース確認（オプション）**
   - [ ] Supabase Dashboard > Table Editor > champion_notesを開く
   - [ ] 作成したノートのレコードが存在する
   - [ ] `user_id`カラムがログインユーザーのUUIDと一致する
   - [ ] `created_at`が現在時刻である

### 期待結果

- ✅ ノートが正常に作成される
- ✅ 作成したノートが一覧に表示される
- ✅ `user_id`が認証済みユーザーのUUIDと一致する
- ✅ エラーが発生しない

### エラーケース

もし失敗した場合、以下を確認:
- [ ] コンソールにエラーログが表示されているか
- [ ] ネットワークタブでAPIリクエストが成功しているか（Status 201）
- [ ] Supabase Authのセッションが有効か（`supabase.auth.getSession()`）
- [ ] RLSポリシーが有効になっているか

## テストケース3: マルチユーザーテスト（RLSポリシー検証）

**目的**: 別のユーザーでログインして、他のユーザーのノートが表示されないことを確認（要件 5.2, 5.3）

### 手順

1. **ユーザーAのノート確認**
   - [ ] 現在ログイン中のユーザーA（user-a@gmail.com）のノート数を記録
   - [ ] ノートの内容を記録（例: Ahri vs Zed）

2. **ログアウト**
   - [ ] Navbarのユーザーアバターをクリック
   - [ ] ドロップダウンメニューから「ログアウト」をクリック
   - [ ] 未認証状態に戻る
   - [ ] Navbarに「ログイン」ボタンが表示される

3. **ユーザーBでログイン**
   - [ ] 「ログイン」ボタンをクリック
   - [ ] 別のGoogleアカウント（user-b@gmail.com）でログイン
   - [ ] ログイン成功後、ユーザーBのアバター画像が表示される

4. **ノート一覧の確認**
   - [ ] `/notes`ページに移動
   - [ ] ノート一覧が空、または以前作成したユーザーBのノートのみが表示される
   - [ ] **重要**: ユーザーAのノート（Ahri vs Zed等）が表示されない

5. **ユーザーBのノート作成**
   - [ ] ユーザーBで新しいノートを作成（例: Yasuo vs Malphite）
   - [ ] ノートが正常に作成される
   - [ ] 一覧にユーザーBのノートのみが表示される

6. **ユーザーAに戻る**
   - [ ] ログアウト
   - [ ] ユーザーAで再ログイン
   - [ ] `/notes`ページに移動
   - [ ] ユーザーAのノート（Ahri vs Zed等）のみが表示される
   - [ ] **重要**: ユーザーBのノート（Yasuo vs Malphite）が表示されない

7. **データベース確認（オプション）**
   - [ ] Supabase Dashboard > Table Editor > champion_notesを開く
   - [ ] ユーザーAとユーザーBのノートが両方存在する
   - [ ] 各ノートの`user_id`が異なるUUIDである
   - [ ] RLSポリシーにより、フロントエンドでは自分のノートのみ取得される

### 期待結果

- ✅ ユーザーAはユーザーAのノートのみ表示される
- ✅ ユーザーBはユーザーBのノートのみ表示される
- ✅ 他のユーザーのノートは一切表示されない
- ✅ RLSポリシーが正しく機能している

### エラーケース

もし他のユーザーのノートが表示された場合:
- [ ] **重大なセキュリティ問題**: RLSポリシーが機能していない
- [ ] Supabase Dashboard > Authentication > Policiesを確認
- [ ] `champion_notes`テーブルのRLSポリシーが有効か確認
- [ ] ポリシーの条件が`auth.uid() = user_id`になっているか確認

## テストケース5: RLSポリシーの直接検証（データベースレベル）

**目的**: データベースレベルでRLSポリシーが正しく機能することを確認（要件 5.2）

### 手順

1. **Supabase SQL Editorを開く**
   - [ ] Supabase Dashboard > SQL Editorを開く

2. **認証済みユーザーとしてクエリ実行**
   - [ ] 以下のSQLを実行（ユーザーAのUUIDを使用）:
   ```sql
   -- ユーザーAのUUIDを確認
   SELECT id, email FROM auth.users WHERE email = 'user-a@gmail.com';
   
   -- ユーザーAのノートを取得（RLSポリシー適用）
   SELECT * FROM champion_notes WHERE user_id = '<ユーザーAのUUID>';
   ```
   - [ ] ユーザーAのノートのみが返される

3. **RLSポリシーの確認**
   - [ ] 以下のSQLを実行:
   ```sql
   -- RLSポリシーの一覧を確認
   SELECT schemaname, tablename, policyname, permissive, roles, cmd, qual
   FROM pg_policies
   WHERE tablename = 'champion_notes';
   ```
   - [ ] 以下のポリシーが存在することを確認:
     - `Users can view their own notes` (SELECT)
     - `Users can create their own notes` (INSERT)
     - `Users can update their own notes` (UPDATE)
     - `Users can delete their own notes` (DELETE)

4. **RLSポリシーの動作確認**
   - [ ] Supabase Dashboard > Table Editor > champion_notesを開く
   - [ ] フィルターなしで全てのノートを表示
   - [ ] 複数のユーザーのノートが存在することを確認
   - [ ] 各ノートの`user_id`が異なることを確認

### 期待結果

- ✅ RLSポリシーが有効になっている
- ✅ 各ユーザーは自分のノートのみアクセスできる
- ✅ `auth.uid() = user_id`の条件が正しく機能している

## テストケース6: エラーハンドリングの確認

**目的**: RLSポリシー違反時のエラーハンドリングを確認（要件 5.4）

### 手順

1. **ログイン**
   - [ ] ユーザーAでログイン

2. **開発者ツールでAPIリクエストを改ざん（高度なテスト）**
   - [ ] ブラウザの開発者ツール > Consoleを開く
   - [ ] 以下のコードを実行して、他のユーザーのノートを取得しようとする:
   ```javascript
   const { createClient } = await import('@supabase/supabase-js');
   const supabase = createClient(
     'YOUR_SUPABASE_URL',
     'YOUR_SUPABASE_ANON_KEY'
   );
   
   // 他のユーザーのUUIDを指定してノートを取得しようとする
   const { data, error } = await supabase
     .from('champion_notes')
     .select('*')
     .eq('user_id', '<他のユーザーのUUID>');
   
   console.log('Data:', data);
   console.log('Error:', error);
   ```
   - [ ] `data`が空配列`[]`である（RLSポリシーによりフィルタリングされる）
   - [ ] エラーは発生しない（RLSは静かにフィルタリングする）

3. **未認証状態でのAPIリクエスト**
   - [ ] ログアウト
   - [ ] 開発者ツール > Consoleで以下を実行:
   ```javascript
   const { createClient } = await import('@supabase/supabase-js');
   const supabase = createClient(
     'YOUR_SUPABASE_URL',
     'YOUR_SUPABASE_ANON_KEY'
   );
   
   // 未認証状態でノートを取得しようとする
   const { data, error } = await supabase
     .from('champion_notes')
     .select('*');
   
   console.log('Data:', data);
   console.log('Error:', error);
   ```
   - [ ] `data`が空配列`[]`である
   - [ ] または、エラーが返される

### 期待結果

- ✅ RLSポリシー違反時、データが返されない
- ✅ エラーが適切にハンドリングされる
- ✅ セキュリティが保たれる

## トラブルシューティング

### 問題1: ノートが作成できない

**症状**: ノート作成時にエラーが発生する

**確認事項**:
- [ ] Supabase Authのセッションが有効か（`supabase.auth.getSession()`）
- [ ] `user_id`が正しく設定されているか
- [ ] RLSポリシーが有効になっているか
- [ ] 外部キー制約が正しく設定されているか（`user_id` → `auth.users(id)`）

**解決方法**:
1. コンソールログを確認
2. Supabase Dashboard > Table Editor > champion_notesでRLSポリシーを確認
3. SQL Editorで手動でINSERTを試行

### 問題2: 他のユーザーのノートが表示される

**症状**: マルチユーザーテストで他のユーザーのノートが表示される

**確認事項**:
- [ ] RLSポリシーが有効になっているか
- [ ] ポリシーの条件が`auth.uid() = user_id`になっているか
- [ ] Supabase Clientが認証済みセッションを使用しているか

**解決方法**:
1. Supabase Dashboard > Authentication > Policiesを確認
2. RLSポリシーを再作成
3. `ALTER TABLE champion_notes ENABLE ROW LEVEL SECURITY;`を実行

### 問題3: 未認証ユーザーがノートページにアクセスできる

**症状**: 未認証状態でもノートページが表示される

**確認事項**:
- [ ] `NotesPage`コンポーネントで認証チェックが実装されているか
- [ ] `useAuth`フックが正しく動作しているか
- [ ] リダイレクト処理が実装されているか

**解決方法**:
1. `frontend/src/app/notes/page.tsx`を確認
2. `useEffect`で認証チェックとリダイレクトを実装
3. ローディング状態を適切に処理
