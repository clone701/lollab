# AIコーディングワークフローガイドライン

## 概要

このステアリングドキュメントは、実践的なAIコーディング手法に基づいたAI支援ソフトウェア開発の体系的なワークフローを定義します。このワークフローにより、構造化されたアプローチを通じて一貫した高品質なコード生成を実現します。

## コア開発ワークフロー

### フェーズ1: プロダクト要求仕様書の作成
**目的**: ユーザーと協力して要件と仕様を具体化する

#### プロセス
1. **ユーザーインタラクション**: ニーズを理解するための反復的な対話を行う
2. **要件収集**: 機能要件と非機能要件を抽出する
3. **仕様書作成**: 包括的な要件ドキュメントを作成する
4. **検証**: ステークホルダーと要件の正確性を確認する

#### 主要ドキュメント
- 参照: 構造とフォーマットのために `docs/product-requirements.md`
- 参照: 一貫した用語のために `docs/glossary.md`
- 出力: specディレクトリの `requirements.md`

#### 品質ゲート
- [ ] すべてのユーザーストーリーに明確な受け入れ基準がある
- [ ] 非機能要件が定量化されている
- [ ] 要件がテスト可能で検証可能である
- [ ] ステークホルダーの承認を得ている

### フェーズ2: 要件からの設計生成
**目的**: 要件を技術設計仕様に変換する

#### プロセス
1. **アーキテクチャ計画**: システムアーキテクチャとコンポーネントを定義する
2. **技術設計**: 詳細な技術仕様を作成する
3. **API設計**: インターフェースとデータモデルを定義する
4. **コンポーネント設計**: 個々のコンポーネントの動作を仕様化する

#### 主要ドキュメント
- 参照: 技術的制約のために `docs/architecture.md`
- 参照: 設計パターンのために `docs/functional-design.md`
- 参照: ファイル構成のために `.kiro/steering/structure.md`
- 出力: specディレクトリの `design.md`

#### 品質ゲート
- [ ] アーキテクチャが要件と整合している
- [ ] 設計が技術的に実現可能である
- [ ] コンポーネントが明確に定義され凝集性がある
- [ ] インターフェースが明確に仕様化されている

### フェーズ3: テスト仕様・テストコードの生成
**目的**: 包括的なテスト生成によりテスト駆動開発をサポートする

#### プロセス
1. **テスト戦略**: テストアプローチとフレームワークを定義する
2. **プロパティベーステスト**: 正確性プロパティを生成する
3. **ユニットテスト生成**: 包括的なユニットテストを作成する
4. **統合テスト計画**: システムレベルのテストを設計する

#### 主要ドキュメント
- 参照: テスト基準のために `docs/development-guidelines.md`
- 参照: テストフレームワークのために `docs/architecture.md`
- 出力: `design.md` のテスト仕様とテストコードファイル

#### 品質ゲート
- [ ] すべての要件に対応するテストがある
- [ ] プロパティベーステストが正確性プロパティをカバーしている
- [ ] テストカバレッジが品質基準（80%以上）を満たしている
- [ ] テストが実行可能でメンテナンス可能である

### フェーズ4: 一貫性のある実装の生成
**目的**: 複数ファイルにわたって正しく機能するコードを生成する

#### プロセス
1. **実装計画**: 設計を実装タスクに分解する
2. **コード生成**: 一貫した動作するコードを生成する
3. **統合**: コンポーネントが正しく連携することを確認する
4. **検証**: 設計に対して実装を検証する

#### 主要ドキュメント
- 参照: コーディング基準のために `docs/development-guidelines.md`
- 参照: 命名規則のために `docs/glossary.md`
- 出力: 実装コードファイルと `tasks.md`

#### 品質ゲート
- [ ] コードが確立されたパターンと規則に従っている
- [ ] すべてのコンポーネントが正しく統合されている
- [ ] 実装が設計仕様を満たしている
- [ ] コードがメンテナンス可能で適切にドキュメント化されている

### フェーズ5: レビューと改善
**目的**: 体系的なフィードバックと改善によりコード品質を確保する

#### プロセス
1. **コードレビュー**: 生成されたコードの体系的なレビュー
2. **品質評価**: 品質メトリクスに対して評価する
3. **フィードバック統合**: 改善提案を取り込む
4. **反復的改善**: 継続的な改善サイクル

#### 主要ドキュメント
- 参照: レビュー基準のために `docs/development-guidelines.md`
- 参照: 品質基準のために `docs/architecture.md`
- 出力: 改善されたコードとドキュメントの更新

#### 品質ゲート
- [ ] コードがすべての品質チェックをパスしている
- [ ] パフォーマンス要件が満たされている
- [ ] セキュリティ要件が満たされている
- [ ] ドキュメントが完全で正確である

## ワークフロー強制ルール

### 必須5フェーズ順次実行
このワークフローは**厳密な順序で必ず**従わなければなりません。フェーズのスキップや順序の変更は禁止です:

1. **プロダクト要求仕様書を作成する** - ユーザーと協力して要件と仕様を具体化する
2. **プロダクト要求仕様書から設計を起こす** - 要件を技術設計仕様に変換する
3. **設計からテスト仕様・テストコードを生成する** - 設計からテスト仕様とテストコードを生成する
4. **一貫性のある実装を生成する** - 複数ファイルにわたって一貫した実装を生成する
5. **レビューと改善を行う** - フィードバックを通じて生成されたコードの品質をレビュー・改善する

### 順次実行要件
- 次のフェーズに進む前に各フェーズを完全に完了させる**こと**
- フェーズ移行前にユーザーの承認・検証を得る**こと**
- フェーズ間の完全なトレーサビリティを維持する**こと**
- フェーズをスキップしたり順序を変えたりしては**ならない**
- 設計とテストフェーズを完了せずに実装に進んでは**ならない**

### 必須ドキュメント参照
- 各フェーズ開始前に関連する `docs/` ファイルを参照・遵守する**こと**
- docsで確立されたパターンとの厳密な一貫性を維持する**こと**
- すべての出力物を参照ドキュメント基準に対して検証する**こと**
- 新しいパターンや手法を導入する際はドキュメントを更新する**こと**

### フェーズ別必須 docs/ ファイル参照:
- **フェーズ1（要件定義）**: `docs/product-requirements.md`, `docs/glossary.md`
- **フェーズ2（設計）**: `docs/architecture.md`, `docs/functional-design.md`, `.kiro/steering/structure.md`
- **フェーズ3（テスト）**: `docs/development-guidelines.md`, `docs/architecture.md`
- **フェーズ4（実装）**: `docs/development-guidelines.md`, `docs/glossary.md`
- **フェーズ5（レビュー）**: `docs/development-guidelines.md`, `docs/architecture.md`

### 品質保証要件
- フェーズ完了前にすべての品質ゲートを満たす**こと**
- フェーズ目標と参照ドキュメントに対して出力物を検証する**こと**
- 開発全体を通じて品質メトリクスを維持する**こと**
- 生成されたすべての成果物が確立された基準に準拠することを確認する**こと**

## フェーズ移行基準

### 要件定義 → 設計
- [ ] 要件が完成し承認されている
- [ ] すべての受け入れ基準がテスト可能である
- [ ] 技術的制約が特定されている

### 設計 → テスト
- [ ] 設計が技術的に健全で完全である
- [ ] すべてのコンポーネントとインターフェースが仕様化されている
- [ ] アーキテクチャの決定がドキュメント化されている

### テスト → 実装
- [ ] テスト仕様が包括的である
- [ ] プロパティベーステストが定義されている
- [ ] テストフレームワークが確立されている

### 実装 → レビュー
- [ ] すべてのコードが生成・統合されている
- [ ] 実装が初期検証をパスしている
- [ ] ドキュメントが更新されている

### レビュー → 完了
- [ ] すべての品質ゲートが満たされている
- [ ] コードレビューのフィードバックが対応済みである
- [ ] 最終検証が成功している
