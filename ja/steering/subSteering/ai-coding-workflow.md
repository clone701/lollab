# AIコーディングワークフローガイドライン

## 概要

このステアリングドキュメントは、実践的なAIコーディング手法に基づく、AI支援ソフトウェア開発の体系的なワークフローを定義します。このワークフローは、構造化されたアプローチを通じて一貫性のある高品質なコード生成を保証します。

## コア開発ワークフロー

### フェーズ1: プロダクト要求仕様書の作成
**目的**: ユーザーとやりとりしながら、要求や要件を具体化する

#### プロセス
1. **ユーザーインタラクション**: ニーズを理解するための反復的な対話
2. **要件収集**: 機能要件と非機能要件の抽出
3. **仕様書作成**: 包括的な要件ドキュメントの作成
4. **検証**: ステークホルダーとの要件精度の確認

#### 主要ドキュメント
- 参照: `docs/product-requirements.md` 構造とフォーマット
- 参照: `docs/glossary.md` 一貫した用語定義
- 出力: 仕様ディレクトリ内の `requirements.md`

#### 品質ゲート
- [ ] すべてのユーザーストーリーに明確な受け入れ条件がある
- [ ] 非機能要件が定量化されている
- [ ] 要件がテスト可能で検証可能である
- [ ] ステークホルダーの承認を得ている

### フェーズ2: プロダクト要求仕様書から設計を起こす
**目的**: 要件を技術的な設計に変換する

#### プロセス
1. **アーキテクチャ計画**: システムアーキテクチャとコンポーネントの定義
2. **技術設計**: 詳細な技術仕様の作成
3. **API設計**: インターフェースとデータモデルの定義
4. **コンポーネント設計**: 個別コンポーネントの動作仕様

#### 主要ドキュメント
- 参照: `docs/architecture.md` 技術的制約
- 参照: `docs/functional-design.md` 設計パターン
- 参照: `docs/repository-structure.md` ファイル構成
- 出力: 仕様ディレクトリ内の `design.md`

#### 品質ゲート
- [ ] アーキテクチャが要件と整合している
- [ ] 設計が技術的に実現可能である
- [ ] コンポーネントが明確に定義され結合性が高い
- [ ] インターフェースが明確に仕様化されている

### フェーズ3: 設計からテスト仕様・テストコードを生成する
**目的**: テスト駆動開発をサポートする包括的なテスト生成

#### プロセス
1. **テスト戦略**: テストアプローチとフレームワークの定義
2. **プロパティベーステスト**: 正確性プロパティの生成
3. **ユニットテスト生成**: 包括的なユニットテストの作成
4. **統合テスト計画**: システムレベルテストの設計

#### 主要ドキュメント
- 参照: `docs/development-guidelines.md` テスト標準
- 参照: `docs/architecture.md` テストフレームワーク
- 出力: `design.md`内のテスト仕様とテストコードファイル

#### 品質ゲート
- [ ] すべての要件に対応するテストがある
- [ ] プロパティベーステストが正確性プロパティをカバーしている
- [ ] テストカバレッジが品質基準を満たしている（80%以上）
- [ ] テストが実行可能で保守可能である

### フェーズ4: 一貫性のある実装を生成する
**目的**: 複数ファイルにわたって正しく動くコードを生成する

#### プロセス
1. **実装計画**: 設計を実装タスクに分解
2. **コード生成**: 一貫性のある動作するコードの生成
3. **統合**: コンポーネントが正しく連携することの確保
4. **検証**: 設計に対する実装の検証

#### 主要ドキュメント
- 参照: `docs/development-guidelines.md` コーディング標準
- 参照: `docs/glossary.md` 命名規則
- 出力: 実装コードファイルと `tasks.md`

#### 品質ゲート
- [ ] コードが確立されたパターンと規約に従っている
- [ ] すべてのコンポーネントが正しく統合されている
- [ ] 実装が設計仕様を満たしている
- [ ] コードが保守可能で適切にドキュメント化されている

### フェーズ5: レビューと改善を行う
**目的**: 体系的なフィードバックと改善を通じてコード品質を確保する

#### プロセス
1. **コードレビュー**: 生成されたコードの体系的レビュー
2. **品質評価**: 品質メトリクスに対する評価
3. **フィードバック統合**: 改善提案の組み込み
4. **反復的改善**: 継続的改善サイクル

#### 主要ドキュメント
- 参照: `docs/development-guidelines.md` レビュー基準
- 参照: `docs/architecture.md` 品質標準
- 出力: 改善されたコードとドキュメント更新

#### 品質ゲート
- [ ] コードがすべての品質チェックを通過している
- [ ] パフォーマンス要件が満たされている
- [ ] セキュリティ要件が満たされている
- [ ] ドキュメントが完全で正確である

## ワークフロー実施ルール

### 必須の5段階順次実行
このワークフローは**必ず**厳密な順次実行で従わなければなりません。フェーズのスキップや順序の変更は許可されません：

1. **プロダクト要求仕様書を作成する** - ユーザーとやりとりしながら、要求や要件を具体化する
2. **プロダクト要求仕様書から設計を起こす** - 要件を技術的な設計仕様に変換する
3. **設計からテスト仕様・テストコードを生成する** - 設計からテスト仕様とテストコードを生成する
4. **一貫性のある実装を生成する** - 複数ファイルにわたって一貫性のある実装を生成する
5. **レビューと改善を行う** - フィードバックを通じて生成コードの品質をレビューし改善する

### 順次実行要件
- 次のフェーズに進む前に各フェーズを**必ず**完全に完了する
- フェーズ移行前にユーザーの承認/検証を**必ず**取得する
- フェーズ間の完全なトレーサビリティを**必ず**維持する
- フェーズのスキップや順序変更は**絶対に**行わない
- 設計とテストフェーズを完了せずに実装に進むことは**絶対に**しない

### 必須のドキュメント参照
- 各フェーズ開始前に関連する`docs/`ファイルからのガイダンスを**必ず**参照し従う
- docs内の確立されたパターンとの厳密な一貫性を**必ず**維持する
- 参考ドキュメント標準に対してすべての出力を**必ず**検証する
- 新しいパターンやアプローチを導入する際にドキュメントを**必ず**更新する

### フェーズ別必須docs/ファイル参照:
- **フェーズ1（要件）**: `docs/product-requirements.md`, `docs/glossary.md`
- **フェーズ2（設計）**: `docs/architecture.md`, `docs/functional-design.md`, `docs/repository-structure.md`
- **フェーズ3（テスト）**: `docs/development-guidelines.md`, `docs/architecture.md`
- **フェーズ4（実装）**: `docs/development-guidelines.md`, `docs/glossary.md`
- **フェーズ5（レビュー）**: `docs/development-guidelines.md`, `docs/architecture.md`

### 品質保証要件
- フェーズ完了前にすべての品質ゲートを**必ず**満たす
- フェーズ目的と参考docsに対する出力を**必ず**検証する
- 開発全体を通じて品質メトリクスを**必ず**維持する
- 生成されるすべての成果物が確立された標準に準拠することを**必ず**保証する

## フェーズ移行基準

### 要件 → 設計
- [ ] 要件が完全で承認されている
- [ ] すべての受け入れ条件がテスト可能である
- [ ] 技術的制約が特定されている

### 設計 → テスト
- [ ] 設計が技術的に健全で完全である
- [ ] すべてのコンポーネントとインターフェースが仕様化されている
- [ ] アーキテクチャ決定がドキュメント化されている

### テスト → 実装
- [ ] テスト仕様が包括的である
- [ ] プロパティベーステストが定義されている
- [ ] テストフレームワークが確立されている

### 実装 → レビュー
- [ ] すべてのコードが生成され統合されている
- [ ] 実装が初期検証を通過している
- [ ] ドキュメントが更新されている

### レビュー → 完了
- [ ] すべての品質ゲートが満たされている
- [ ] コードレビューフィードバックが対応されている
- [ ] 最終検証が成功している
