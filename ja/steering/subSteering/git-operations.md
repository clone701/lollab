# Git操作ガイドライン

## 概要

このステアリング文書は、変更のコミットとプルリクエスト作成のための標準化されたGitワークフローを定義します。この文書は、ユーザーがGit関連の操作を要求した場合にのみ参照されるべきです。

## 標準ブランチ作成ワークフロー

ユーザーが「XXブランチを作成して」と要求した場合、この正確な順序に従ってください：

### ステップ1: 最新の変更を取得
```bash
git fetch origin
```

### ステップ2: mainブランチに切り替え
```bash
git checkout main
```

### ステップ3: 最新のmainをプル
```bash
git pull origin main
```

### ステップ4: mainから新しいブランチを作成
```bash
git checkout -b [ブランチ名]
```

**完全なコマンドシーケンス：**
```bash
git fetch origin ; git checkout main ; git pull origin main ; git checkout -b [ブランチ名]
```

## 標準コミットワークフロー

ユーザーが「コミットして」と要求した場合、この正確な順序に従ってください：

### ステップ1: 変更をステージングしステータス確認
```bash
git add . ; git status
```

**出力を注意深く確認してください：**
- 意図したファイルがすべてステージングされていることを確認
- 予期しないファイルがないかチェック
- 機密情報が含まれていないことを確認

### ステップ2: 変更をコミット
ステップ1で問題がない場合のみ進行：
```bash
git commit -m "説明的なコミットメッセージ"
```

**コミットメッセージ形式：**
- 形式を使用: `[ブランチ名] type: 日本語説明`
- 例: `kiro-1 feat: Git操作ガイドラインを追加`
- タイプ: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`
- 説明は日本語で簡潔だが説明的に保つ
- トレーサビリティのためブランチ名プレフィックスを含める

### ステップ3: プルリクエスト作成
コミット成功後、プルリクエストを作成：
```bash
git push origin [現在のブランチ名]
```

その後、Gitプラットフォームのインターフェースを通じてプルリクエストを作成するようユーザーを案内。

## エラーハンドリング

### git addが失敗した場合：
- ファイル権限を確認
- リポジトリステータスを確認
- ユーザーに具体的なエラーを報告

### git statusで問題が表示された場合：
- コミットを**進行しない**
- ユーザーに問題を報告
- 続行前にユーザーの確認を待つ

### git commitが失敗した場合：
- Git設定を確認 (user.name, user.email)
- コミットメッセージ形式を確認
- ユーザーに具体的なエラーを報告

## Git設定要件

操作前にGitが適切に設定されていることを確認：
```bash
git config user.name "LoL Lab Developer"
git config user.email "developer@lollab.local"
```

## ブランチ管理

### 現在のブランチ確認
操作前に常に現在のブランチを確認：
```bash
git branch
```

### ブランチ命名規則
- 説明的な名前を使用: `feature/description`, `fix/issue-name`, `kiro-1`など
- スペースと特殊文字を避ける
- ハイフン付きの小文字を使用

## セキュリティ考慮事項

### ステージング前 (git add)：
- 機密情報を**絶対に**コミットしない：
  - APIキー、パスワード、トークン
  - 個人情報
  - データベース認証情報
  - 環境ファイル (.env)

### ファイルレビューチェックリスト：
- [ ] ステージングされたファイルに機密データがない
- [ ] すべてのファイルが意図的に含まれている
- [ ] 一時ファイルやキャッシュファイルがない
- [ ] ドキュメントが最新である

## プルリクエストガイドライン

### PRタイトル形式：
- コミットメッセージと同じ形式を使用: `[ブランチ名] type: 日本語説明`
- 日本語で説明的で簡潔にする

### PR説明に含めるべき内容：
- 変更の要約
- 変更の理由
- 破壊的変更
- 実行されたテスト

## 自動化ルール

### ユーザーが「XXブランチを作成して」と言った場合：
1. git fetch origin ; git checkout main ; git pull origin main ; git checkout -b [ブランチ名] を**必ず**実行
2. ブランチ作成が成功したことを**必ず**確認
3. 現在のブランチが新しく作成されたブランチであることを**必ず**確認
4. ユーザーにブランチ作成成功を**通知すべき**

### ユーザーが「コミットして」と言った場合：
1. git add . ; git status を**必ず**実行
2. 出力を注意深くレビューし問題がないか**必ず**確認
3. ステータスがクリーンな場合のみ**必ず**進行
4. 適切なメッセージでgit commit を**必ず**実行
5. プルリクエスト作成を**提案すべき**

### エラー応答：
- ユーザーに日本語でエラーを報告
- 具体的なエラー詳細を提供
- 修正アクションを提案
- ユーザーの確認なしに進行しない

## 開発ワークフローとの統合

このGitワークフローは5段階AIコーディングワークフローと統合：
- 各フェーズ完了後に使用
- 段階的な進捗をコミット
- 主要な変更にはフィーチャーブランチを作成
- レビュー後のみmainにマージ

## 環境固有の注意事項

### Windows PowerShell：
- 必要に応じて完全なGitパスを使用: `"C:\Program Files\Git\bin\git.exe"`
- PATH環境変数を設定: `$env:PATH += ";C:\Program Files\Git\bin"`
- PowerShellコマンド解析を慎重に処理

### コマンド実行：
- 操作前に常にGitが利用可能であることを確認
- パスの問題を適切に処理
- 日本語で明確なエラーメッセージを提供
