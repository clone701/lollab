# 要件定義書

## はじめに

LoL Labは、League of Legendsのチャンピオン戦略アプリケーションで、プレイヤーがサモナー分析、チャンピオンノート作成、試合履歴レビューを通じてゲームプレイを向上させることを支援します。このシステムにより、プレイヤーは対戦相手を調査し、カウンター戦略を文書化し、異なるチャンピオンマッチアップでの学習進捗を追跡できます。

## 用語集

- **System**: LoL Labウェブアプリケーション
- **Summoner**: League of Legendsプレイヤーアカウント
- **Champion**: League of Legendsでプレイ可能なキャラクター
- **Matchup**: プレイヤーチャンピオン対敵チャンピオンの特定の組み合わせ
- **Note**: 特定のマッチアップに対するユーザー作成の戦略ドキュメント
- **Riot_API**: League of Legendsデータ用の公式Riot Games API
- **Cache_System**: APIレスポンス用のRedisベースキャッシュレイヤー
- **User**: アプリケーションを使用する認証済みプレイヤー

## 要件

### 要件1: サモナー検索と分析

**ユーザーストーリー:** League of Legendsプレイヤーとして、サモナープロフィールを検索・分析したいので、対戦相手の強みを理解し、カウンター戦略を準備できます。

#### 受け入れ基準

1. ユーザーがサモナー名とリージョンを入力したとき、SystemはRiot_APIにクエリを送信し、サモナープロフィールデータを返すものとする
2. サモナーデータが取得されたとき、Systemは現在のランク、ティア、リーグポイントを表示するものとする
3. サモナー情報を表示するとき、Systemは最新10試合のランク戦でチャンピオン、K/D/A、CS、ダメージ統計を表示するものとする
4. サモナーが見つからないとき、Systemは説明的なエラーメッセージを返すものとする
5. Riot_APIが利用できないとき、Systemはエラーを適切に処理し、ユーザーに通知するものとする

### 要件2: チャンピオンノート管理

**ユーザーストーリー:** プレイヤーとして、チャンピオン固有のノートを作成・管理したいので、異なるマッチアップに対する効果的な戦略とカウンタープレイを文書化できます。

#### 受け入れ基準

1. ユーザーがチャンピオンマッチアップを選択したとき、Systemはノートを作成または編集するインターフェースを提供するものとする
2. ノートを作成するとき、Systemはユーザーがルーン、サモナースペル、開始アイテム、戦略テキストを指定できるようにするものとする
3. ノートが保存されるとき、Systemはユーザー関連付けでデータベースに永続化するものとする
4. 既存のノートを表示するとき、Systemは保存されたすべての情報を整理された形式で表示するものとする
5. ユーザーがノートを削除するとき、Systemはデータベースから削除し、インターフェースを更新するものとする

### 要件3: データ永続化と取得

**ユーザーストーリー:** ユーザーとして、ノートと設定がセッション間で保存・アクセス可能であってほしいので、包括的な戦略知識ベースを構築できます。

#### 受け入れ基準

1. ユーザーがノートを作成または更新するとき、Systemは適切なユーザー関連付けでchampion_notesテーブルに保存するものとする
2. ユーザーがログインするとき、Systemは保存されたすべてのノートを取得し、適切に表示するものとする
3. ユーザーがノートを検索するとき、Systemはチャンピオンまたはマッチアップ基準で結果をフィルタリングするものとする
4. ローカルストレージデータが存在するとき、Systemはブラウザセッション間で最近の検索とピン留めチャンピオンを維持するものとする
5. データベース操作が失敗するとき、Systemはエラーを適切に処理し、データ整合性を維持するものとする

### 要件4: Riot API統合とキャッシュ

**ユーザーストーリー:** システム管理者として、適切なキャッシュによる効率的なAPI使用を望むので、レート制限を尊重しながらアプリケーションが良好に動作します。

#### 受け入れ基準

1. Riot_APIリクエストを行うとき、SystemはAPI違反を避けるため適切なレート制限を実装するものとする
2. サモナーデータが要求されるとき、Cache_SystemはAPIコールを削減するため5分間レスポンスを保存するものとする
3. キャッシュされたデータが存在し有効であるとき、Systemは新しいAPIコールの代わりにキャッシュされた結果を返すものとする
4. APIレート制限を超過するとき、Systemはリクエストをキューに入れ、指数バックオフで再試行するものとする
5. APIキーが管理されるとき、Systemは環境変数に安全に保存するものとする

### 要件5: ユーザーインターフェースと体験

**ユーザーストーリー:** ユーザーとして、直感的でレスポンシブなインターフェースを望むので、任意のデバイスでアプリケーションを効率的にナビゲートし使用できます。

#### 受け入れ基準

1. モバイルデバイスでアプリケーションにアクセスするとき、Systemは画面サイズに適応するレスポンシブデザインを提供するものとする
2. ページ間をナビゲートするとき、Systemは一貫したUIパターンとローディング状態を維持するものとする
3. データテーブルを表示するとき、Systemは適切なソートとフィルタリング機能を実装するものとする
4. エラーが発生するとき、Systemは推奨アクションを含むユーザーフレンドリーなエラーメッセージを表示するものとする
5. データをロードするとき、Systemはシステム状態をユーザーに通知する適切なローディングインジケーターを表示するものとする

### 要件6: 認証とセキュリティ

**ユーザーストーリー:** ユーザーとして、安全な認証とデータ保護を望むので、個人的なノートと設定がプライベートに保たれます。

#### 受け入れ基準

1. ユーザーが保護された機能にアクセスしようとするとき、SystemはGoogle OAuth認証を要求するものとする
2. ユーザーが正常にログインするとき、Systemはapp_usersテーブルでプロフィールを作成または更新するものとする
3. ユーザーデータにアクセスするとき、Systemはユーザーが自分のノートのみを表示・変更できることを保証するものとする
4. 機密データを処理するとき、Systemは適切な入力検証とサニタイゼーションを実装するものとする
5. APIキーを保存するとき、Systemは安全な環境変数管理を使用するものとする

### 要件7: パフォーマンスと信頼性

**ユーザーストーリー:** ユーザーとして、高速で信頼性の高いアプリケーションパフォーマンスを望むので、チャンピオン選択とゲームプレイ準備中に迅速に情報にアクセスできます。

#### 受け入れ基準

1. ホームページをロードするとき、Systemは通常の条件下で2秒以内にコンテンツを表示するものとする
2. サモナーを検索するとき、Systemは5秒以内に結果を返すか、タイムアウトフィードバックを提供するものとする
3. データベースが負荷状態にあるとき、Systemはノート操作で3秒未満のレスポンス時間を維持するものとする
4. 外部APIが失敗するとき、Systemはフォールバック動作と適切な劣化を提供するものとする
5. 同時ユーザーを処理するとき、Systemはデータ破損なしにパフォーマンスを維持するものとする

### 要件8: データ検証とエラー処理

**ユーザーストーリー:** システムとして、堅牢なデータ検証とエラー処理を望むので、アプリケーションが安定し、ユーザーに意味のあるフィードバックを提供します。

#### 受け入れ基準

1. ユーザーがサモナー名を入力するとき、Systemは形式とリージョン互換性を検証するものとする
2. チャンピオンノートを保存するとき、Systemはすべての必須フィールドとデータ型を検証するものとする
3. APIレスポンスが不正な形式であるとき、Systemは解析エラーを適切に処理するものとする
4. データベース制約が違反されるとき、Systemはユーザーに具体的なエラーメッセージを提供するものとする
5. 予期しないエラーが発生するとき、Systemはユーザーフレンドリーなメッセージを表示しながら、デバッグ用の詳細情報をログに記録するものとする

### 要件9: 検索と発見機能

**ユーザーストーリー:** ユーザーとして、保存されたノートを効率的に検索・発見したいので、ゲームプレイ準備中に関連する戦略を迅速に見つけることができます。

#### 受け入れ基準

1. ノートを検索するとき、Systemは自分のチャンピオン、敵チャンピオン、またはその両方でフィルタリングを許可するものとする
2. 検索結果を表示するとき、Systemは作成日とマッチアップ情報を含むノートプレビューを表示するものとする
3. 検索結果が見つからないとき、Systemは代替検索用語を提案するか、人気のマッチアップを表示するものとする
4. ノートを閲覧するとき、Systemは日付、チャンピオン、またはマッチアップ頻度によるソートオプションを提供するものとする
5. ノートリストを表示するとき、Systemは大きな結果セットに対してページネーションを実装するものとする

### 要件10: テストと品質保証

**ユーザーストーリー:** 開発者として、包括的なテストカバレッジを望むので、アプリケーションがすべての機能で信頼性と正確性を維持します。

#### 受け入れ基準

1. APIエンドポイントを実装するとき、Systemはすべての CRUD操作に対してユニットテストを含むものとする
2. 外部APIと統合するとき、Systemはモックレスポンスを使用した統合テストを含むものとする
3. ユーザー入力を検証するとき、Systemはエッジケースに対してプロパティベーステストを含むものとする
4. UIコンポーネントをテストするとき、Systemは適切なレンダリングとユーザーインタラクションを検証するものとする
5. 変更をデプロイするとき、Systemはすべてのテストを実行し、テストが失敗した場合はデプロイを防ぐものとする
